# 旋转特征统一结构设计

## 1. 目标

本文在前置调研文档 [旋转特征跨系统参数调研_20260329.md](E:/MyProject/CADExchange/doc/旋转特征跨系统参数调研_20260329.md) 的基础上，提出 `CADExchange` 中 `Revolve` 的统一数据结构设计方案，并明确它与 `CExtrude` 的共用抽象和阶段性实现边界。

本文是设计文档，不直接包含实现代码。

## 2. 设计目标

本次设计需要同时满足以下目标：

1. `Revolve` 不再是孤立模型，尽量复用 `CExtrude` 已有抽象。
2. 支持后续逐步覆盖 SolidWorks、Creo、UG/NX 的真实 revolve 语义。
3. 允许分阶段实施，第一阶段先完成 unified model 和序列化改造，不强行一次做完所有 CAD 桥接。
4. 尽量减少 schema 再次推翻的概率。

## 3. 当前问题

当前 [UnifiedFeatures.h](E:/MyProject/CADExchange/core/UnifiedFeatures.h) 中：

- `CExtrude` 有：
  - `profileSketchID`
  - `direction`
  - `endCondition1 / endCondition2`
  - `operation`
  - `draft`
  - `thinWall`
- `CRevolve` 只有：
  - `profileSketchID`
  - `axis`
  - `angleKind`
  - `primaryAngle`
  - `secondaryAngle`

问题有三类：

### 3.1 语义能力不对齐

`CRevolve` 缺少：

- `operation`
- `thinWall`
- `reference-based extent`
- `offset`
- `helper point`

这使得它比 `CExtrude` 更弱，也无法承载跨系统 revolve 的主流能力。

### 3.2 结构抽象不统一

`CExtrude` 使用：

- `direction + endCondition1 + endCondition2`

`CRevolve` 使用：

- `axis + angleKind + primaryAngle + secondaryAngle`

这导致 builder、accessor、validator、serializer 都无法共享实现思路。

### 3.3 当前 end condition 命名过于 extrude-specific

`ExtrudeEndCondition` 中的很多命名是拉伸视角：

- `depth`
- `UP_TO_FACE`
- `UP_TO_VERTEX`
- `SYMMETRIC`（统一承载原先 `MID_PLANE` 语义）
- `THROUGH_ALL_BOTH_SIDES`

直接拿它给 revolve 复用，语义会变得别扭，也不方便覆盖 NX 的 limit 体系。

## 4. 设计原则

### 4.1 统一“轮廓驱动特征”

Extrude 和 Revolve 都是“由轮廓驱动生成体/面”的特征，因此应优先共享：

- profile
- operation
- thin
- extent

### 4.2 区分“驱动方式”和“终止方式”

Extrude 与 Revolve 的核心差别不在终止条件，而在驱动方式：

- Extrude: 线性方向
- Revolve: 旋转轴

因此：

- `direction / axis` 应作为特征差异
- `extent` 应作为共用抽象

### 4.3 先做可扩展的共用语义桶

第一阶段不要把所有 CAD 特有项都塞进 `CRevolve` 主体，而是先搭出稳定的“共用能力桶”：

- profile
- operation
- extent
- thin
- axis

后续再扩展：

- body kind
- scope
- offset subtype
- helper point
- trim/surface

## 5. 建议的新结构分层

## 5.1 共享基类：轮廓驱动特征

建议新增：

```cpp
struct CProfiledFeatureBase : public CFeatureBase {
  std::string profileSketchID;
  BooleanOp operation = BooleanOp::BOSS;
  std::optional<ThinWallOption> thinWall;
};
```

### 目的

- 把 `profileSketchID`
- `operation`
- `thinWall`

从 `CExtrude` 和 `CRevolve` 中抽到一层共享基类。

### 原因

这三项对 extrude / revolve 都是自然共性：

- 两者都需要轮廓
- 两者都有 add/remove/boolean 语义
- 两者都可能有 thin feature

## 5.2 共享终止条件：SweepExtent

建议新增通用结构：

```cpp
struct SweepExtent {
  enum class Type {
    VALUE,
    SYMMETRIC,
    THROUGH_ALL,
    UP_TO_NEXT,
    UP_TO_ENTITY,
    UP_TO_EXTENDED,
    THRU_POINT,
    UNKNOWN
  } type = Type::UNKNOWN;

  double value = 0.0;
  double offset = 0.0;
  bool hasOffset = false;
  bool isFlip = false;
  bool isFlipMaterialSide = false;
  std::shared_ptr<CRefEntityBase> referenceEntity;
  std::optional<CPoint3D> helperPoint;
};
```

### 字段说明

- `type`
  - 表达终止方式
- `value`
  - 对于 extrude 通常是长度
  - 对于 revolve 通常是角度
- `offset`
  - 终止偏置
- `hasOffset`
  - 是否启用偏置
- `isFlip`
  - 终止方向反转
- `isFlipMaterialSide`
  - 材料侧反转
- `referenceEntity`
  - up-to / to reference 的参照
- `helperPoint`
  - 多解消歧位置

### 为什么不直接复用 `ExtrudeEndCondition`

因为 `ExtrudeEndCondition` 的 `Type` 命名已经严重绑定拉伸语义，不适合作为更高层统一抽象。

新增 `SweepExtent` 后：

- `ExtrudeEndCondition` 可以退场
- `Revolve` 也不需要再保留 `AngleKind`

## 5.3 Extrude 新结构

建议改成：

```cpp
struct CExtrude : public CProfiledFeatureBase {
  CVector3D direction = {0, 0, 1};
  SweepExtent extent1;
  std::optional<SweepExtent> extent2;
  std::optional<DraftOption> draft;

  CExtrude() { featureType = FeatureType::Extrude; }
};
```

### 解释

- `direction` 仍然是 Extrude 的特有驱动参数
- `extent1 / extent2` 替代 `endCondition1 / endCondition2`
- `draft` 继续保留为 extrude 特有扩展

## 5.4 Revolve 新结构

建议改成：

```cpp
struct CRevolveAxis {
  std::string referenceLocalID;
  std::shared_ptr<CRefEntityBase> referenceEntity;
  CPoint3D origin;
  CVector3D direction;
};

struct CRevolve : public CProfiledFeatureBase {
  CRevolveAxis axis;
  SweepExtent extent1;
  std::optional<SweepExtent> extent2;

  CRevolve() { featureType = FeatureType::Revolve; }
};
```

### 解释

- `axis` 是 Revolve 的特有驱动参数
- `extent1 / extent2` 统一表达单向、双向、对称和参考终止

这样之后：

- 单向 revolve = 只设置 `extent1`
- 双向 revolve = 同时设置 `extent1 + extent2`
- 对称 revolve = `extent1.type = SYMMETRIC`

## 6. 与现有 CExtrude 成员的复用关系

### 6.1 可以直接复用

以下现有成员或结构可以直接复用：

- `BooleanOp`
- `ThinWallOption`
- `DraftOption`
- `profileSketchID`

### 6.2 建议泛化后复用

以下结构不应原封不动保留，而应升级：

- `ExtrudeEndCondition` -> `SweepExtent`

### 6.3 不建议继续保留的 revolve 特有表达

以下字段建议逐步淘汰：

- `CRevolve::AngleKind`
- `CRevolve::primaryAngle`
- `CRevolve::secondaryAngle`

原因：

- 只能表达角度，不支持到参考
- 不能覆盖 NX limits
- 与 extrude 的双侧终止风格不统一

## 7. 关于 BooleanOp 的设计建议

当前 [UnifiedFeatures.h](E:/MyProject/CADExchange/core/UnifiedFeatures.h) 中：

```cpp
enum class BooleanOp { BOSS, CUT, MERGE };
```

这对当前 extrude 勉强可用，但对于 future revolve / NX 不够。

但本轮设计约束已经明确：

- `BooleanOp` 暂时**不扩展**

因此第一阶段保持原样：

```cpp
enum class BooleanOp {
  BOSS,
  CUT,
  MERGE
};
```

### 当前取舍

- 先完成 `Revolve` 与 `Extrude` 的结构统一
- 暂不把 NX 的 `INTERSECT/SEW` 映射进 unified 层
- 后续如要支持更宽的 boolean 语义，再单独起一轮枚举升级与桥接改造

## 8. 关于 ThinWallOption 的建议

当前 `ThinWallOption`：

```cpp
struct ThinWallOption {
  double thickness = 0.0;
  bool isOneSided = true;
  bool isOutward = false;
  bool isCovered = false;
};
```

### 当前约束

本轮设计约束已经明确：

- `ThinWallOption` 暂时**不扩展**

因此第一阶段直接复用现有结构，不增加：

- 双侧不同厚度
- material side 枚举
- start/end thin reference

### 当前取舍

- 先统一 revolve 对 thin 的承载能力
- 不追求 SW/Creo thin revolve 的 full fidelity
- 后续如果需要，再把 `ThinWallOption` 升级为更通用的 `ThinFeatureOption`

## 9. 关于 profile 的建议

当前仍建议先保留：

- `profileSketchID`

并且本轮设计约束已经明确：

- `profile contour` 暂时**不扩展**

### 原因

- 当前仓库的 builder/accessor/serializer 都建立在 `profileSketchID` 上
- 引入 contours 会显著扩大改造面
- `Revolve` 的统一结构改造与 contour 模型改造属于两个正交主题
- 先完成结构统一，再处理 SW contour / NX section 细粒度选择

### 限制

这一阶段 unified 层仍然只能表达：

- “引用哪个草图”

不能表达：

- 草图中的哪个 contour
- 多 contour 参与方式
- section rule / section object 细节

## 10. 迁移说明

## 10.1 内存模型迁移

建议迁移顺序：

1. 新增 `CProfiledFeatureBase`
2. 新增 `SweepExtent`
3. `CExtrude` 改为继承 `CProfiledFeatureBase`
4. `CRevolve` 改为继承 `CProfiledFeatureBase`
5. 删除旧 revolve 字段，不保留 XML 兼容窗口

## 10.2 序列化

### UnifiedSerialization

建议直接升级为新 schema，不做长期双写。

### TinyXMLSerializer

建议直接切到新 schema：

1. `SaveRevolve` 只写新字段
2. `LoadRevolve` 只接受新字段
3. 不再兼容 `AngleKind / PrimaryAngle / SecondaryAngle`

## 10.3 Builder / Accessor

`RevolveBuilder` 保留 `SetAngle / SetTwoWayAngle / SetSymmetricAngle` 作为语法糖，但底层只写新字段：

- `SetAngle(double)` -> 写入 `extent1`
- `SetTwoWayAngle(double, double)` -> 写入 `extent1 + extent2`
- `SetSymmetricAngle(double)` -> 写入 `extent1.type = SYMMETRIC`

随后新增通用接口：

- `SetExtent1(const SweepExtent&)`
- `SetExtent2(const SweepExtent&)`
- `SetOperation(BooleanOp)`
- `SetThinWall(...)`

`RevolveAccessor` 同理，直接暴露新的共享 extent 读取接口：

- 新增：
  - `GetOperation()`
  - `HasThinWall()`
  - `GetExtentType1()`
  - `GetExtentValue1()`
  - `GetReference1()`
  - `HasExtent2()`

## 11. 校验规则建议

`ModelValidator` 需要从“校验 angle”改成“校验 extent”。

### Revolve 校验建议

- `profileSketchID` 非空
- `axis.direction` 非零
- `operation` 合法
- `extent1.type != UNKNOWN`
- 若 `extent1.type == VALUE`，则 `value > 0`
- 若 `extent1.type == UP_TO_ENTITY`，则 `referenceEntity != nullptr`
- 若存在 `extent2`，同样校验
- 若 `thinWall` 存在，则 `thickness > 0`

### Extrude 校验建议

- 用同一套 `SweepExtent` 校验逻辑
- 额外保留对 `direction` 非零的校验

## 12. 第一阶段范围（推荐）

推荐第一阶段只做以下内容：

1. 统一数据结构
2. 序列化升级
3. builder/accessor/validator 更新
4. old revolve angle schema 不再读取

第一阶段暂不做：

- contour 集合建模
- body scope 完整建模
- helper point 的 CAD 端实现
- NX intersect/sew 的完整桥接
- surface/trim revolve 的统一模型

## 13. 第二阶段范围

第二阶段再考虑：

- `BooleanOp` 扩容
- `body kind`
- `scope / target bodies`
- `helperPoint`
- `surface/trim`
- `profile selection / contour`
- `ThinWallOption` 扩展

## 14. 建议的最终方向

综合当前仓库状态与跨系统需求，推荐采用：

- `CProfiledFeatureBase`
- `SweepExtent`
- `CExtrude/CRevolve` 共用 `profile + operation + thin + extent`
- `direction/axis` 分别作为各自特有驱动参数

这条路线的优点是：

- 与现有 `CExtrude` 最大程度统一
- 可以收敛 builder/accessor/serializer/validator 的实现风格
- 允许先完成架构升级，再逐步扩展 CAD 保真度
