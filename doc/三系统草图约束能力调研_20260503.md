# 三系统草图约束能力调研

日期：2026-05-03

## 1. 目的

本文整理 SolidWorks、Creo、UG/NX 三个系统中“草图约束”的官方 API 能力、当前仓库已验证/已实现能力，以及对 `CADExchange` 统一结构设计的直接影响。

重点区分三件事：

1. **官方 API 理论可表达什么**
2. **本仓库当前桥接代码实际上已经落了什么**
3. **哪些约束适合作为 CADExchange 的公共语义，哪些应当作为扩展语义**

- 写入 relation：`pSketchRelationMgr->IAddRelation(...)`

- [SwLibs/demo/SwLibs/src/SketchInfo.cpp](E:\MyProject\SwLibs\demo\SwLibs\src\SketchInfo.cpp:500)

这说明 **SW 从官方 API 角度最适合做“丰富统一语义”的信息源**。

### 3.5 当前 `SwLibs/read_write` 正式桥接现状

当前正式桥接实现并没有真正把 SW 草图 relation 读写接到 `CADExchange::CSketchConstraint`。写侧主要依赖：

- `AddToDB = TRUE`
- 几何端点坐标完全一致
- 让 SW 把同坐标端点视为同一个拓扑顶点

见：

- [SwLibs/read_write/src/internal/feature_sketch.cpp](E:\MyProject\SwLibs\read_write\src\internal\feature_sketch.cpp:281)

关键注释原文含义：

- 当前模式下不依赖显式 `Coincident`
- 对 SW->SW round-trip 或单位归一化后的跨 CAD 数据，依赖几何重合即可闭合轮廓

结论：

- **SW 官方能力很强**
- **但本仓库正式桥接层当前几乎没有把这些 relation 显式进出统一层**

### 4.3 官方创建接口特征

Creo 的 section 约束不是简单“给两个 local id 就行”，而是典型的：

1. 先拿到 `ProSection` 中具体实体/端点/局部选择
2. 组织 `ProSelection[]`
3. 调 `ProSectionConstraintCreate(...)`

这意味着对统一层来说，Creo 写侧比 SW/NX 更依赖：

- 端点级别定位
- selection 语义
- section 求解状态
- autodimension / alignment / regenerate 的配套流程

### 4.4 当前 `CreoLibs/read_write` 正式桥接现状

当前仓库正式实现已验证并落地的约束只有 `PRO_CONSTRAINT_SAME_POINT`。

核心代码在：

- [CreoLibs/read_write/src/internal/feature_sketch.cpp](E:\MyProject\CreoLibs\read_write\src\internal\feature_sketch.cpp:2092)

当前流程是：

1. 对已创建 section 图元收集端点
2. 按端点几何距离容差配对
3. 用 `ProSectionEntityGetSelected(...)` 取端点选择
4. 调 `ProSectionConstraintCreate(..., PRO_CONSTRAINT_SAME_POINT, ...)`

这条路径已经在仓库研究记录中闭环：

- `PRO_CONSTRAINT_SAME_POINT` 可用
- 但强依赖 Intent Manager 与 section 选择语义
- 如果错误地“按段序而非真实几何重合”去加约束，会直接扭曲轮廓

### 4.5 对统一建模的含义

Creo 官方枚举看起来不弱，但本仓库当前验证层面仍集中在：

- 点重合
- 以及与 autodimension/alignment/references 的配套求解

因此，对 Creo 不应把“官方存在某枚举”直接视为“当前桥接已可稳定支持”。

## 6. 三系统对比总结

### 6.1 官方 API 层面

如果只看官方 API：

- **SW**：relation 枚举最丰富，读写都强，适合做“高信息量源头”
- **Creo**：section 约束枚举较完整，但强依赖 selection 和 section 求解语义
- **UG/NX**：高层 API 清晰，几何约束和尺寸约束分层明确，类型要求最严格

### 6.2 当前仓库正式桥接层面

如果只看当前 `read_write` 正式实现：

- **SW**：主要依赖几何重合和 `AddToDB`，尚未把 relation 系统正式接入统一层
- **Creo**：已明确实现 `PRO_CONSTRAINT_SAME_POINT`
- **UG/NX**：已明确实现 `CreateCoincidentConstraint`

也就是说，当前三端真正稳定落地的公共交集非常小，基本只有：

- `COINCIDENT`

### 6.3 与 CADExchange 当前枚举的差距

`CADExchange` 统一层已经预留：

- `HORIZONTAL`
- `VERTICAL`
- `CONCENTRIC`
- `TANGENT`
- `EQUAL`
- `PARALLEL`
- `PERPENDICULAR`
- `DISTANCE`
- `ANGLE`
- `RADIUS`
- `DIAMETER`
- `SYMMETRIC`
- `MIDPOINT`
- `COLLINEAR`
- `FIXED`

但这些约束在三端桥接中的成熟度并不一致：

- SW：官方上几乎都能做
- Creo：不少约束官方有枚举，但仓库未验证
- NX：不少约束官方有明确接口，但仓库未落地

因此当前统一层更像“目标能力”，而不是“已验证最小公共子集”。

## 7. 对统一结构设计的建议

### 7.1 建议拆成“公共层 + 扩展层”

建议把统一结构按能力成熟度分层：

#### P0 公共层

- `COINCIDENT`

这是当前三端最接近可稳定 round-trip 的公共交集。

#### P1 常见几何层

- `HORIZONTAL`
- `VERTICAL`
- `TANGENT`
- `CONCENTRIC`
- `PARALLEL`
- `PERPENDICULAR`
- `EQUAL`

这层适合作为统一层标准语义继续推进，但实现时必须标注各系统支持状态。

#### P2 尺寸与高级关系层

- `DISTANCE`
- `ANGLE`
- `RADIUS`
- `DIAMETER`
- `SYMMETRIC`
- `MIDPOINT`
- `FIXED`
- `PIERCE`
- `COLLINEAR`
- `ARCLENGTH`
- `EQUAL_CURVATURE`

这层不建议继续压成一个单独 `DIMENSIONAL`。

### 7.2 `CSketchConstraint` 当前已升级结果

截至本次调研落地后，`CADExchange` 已经把草图约束核心结构升级为：

- `type`
- `refs`
- `value`

其中：

- `refs` 使用 `SketchConstraintRef`
- `SketchConstraintRef` 支持：
  - 草图内部图元 `SketchEntity`
  - 外部引用实体 `ExternalReference`
  - `Whole / Start / End / Center / Midpoint`

这意味着统一层已经不再局限于：

- `entityLocalIDs`
- `dimensionValue`

而是能够表达：

- 线端点到线端点的重合
- 线到外部基准面/边/轴的尺寸或几何约束
- 圆心、中点等子实体级约束落点

### 7.3 尺寸约束不要再只用 `DIMENSIONAL`

这条建议已经在 `CADExchange` 当前实现中落实。尺寸约束已拆成独立类型，而不是继续用单一 `DIMENSIONAL` 容纳所有内容。当前至少区分：

- `DISTANCE`
- `ANGLE`
- `RADIUS`
- `DIAMETER`

否则：

- SW 原生语义会被压扁
- NX 的专门尺寸接口无法自然映射
- Creo 的 autodim / alignment / regenerate 也很难做出精确回放

## 8. 当前结论

截至 2026-05-03，本仓库可确认的结论是：

1. `CADExchange` 统一层当前枚举大于三端正式桥接能力。
2. SolidWorks 官方 relation 能力最强，适合做统一模型设计时的“语义上界”。
3. Creo 官方 section 约束种类不少，但仓库当前正式闭环只验证了 `PRO_CONSTRAINT_SAME_POINT`。
4. UG/NX 官方几何和尺寸约束 API 都很丰富，但仓库当前正式落地只用了 `CreateCoincidentConstraint`。
5. 当前若追求三端稳定公共语义，最稳妥的公共交集仍然是 `COINCIDENT`。
6. `CADExchange` 统一模型层已经完成：
   - 尺寸约束从粗粒度 `DIMENSIONAL` 拆细
   - 约束参与对象从裸 `localID` 升级为 `SketchConstraintRef`
7. 下一步更实际的重点不再是 core 结构，而是三端桥接如何逐步把原生约束真正读写到这些统一字段中。
