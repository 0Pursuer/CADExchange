# 三系统特征生成与统一模型映射对照

> 本文档解决一个核心问题：**三个 CAD 系统（SW / Creo / UG）在特征语义上存在根本差异，`CADExchange` 的统一层（`UnifiedModel`）是如何定义中间结构以容纳这种差异，以及读/写时各自做了哪些转换和计算的。**

## 1. 总体说明

### 1.1 为什么要统一层

SolidWorks、Creo、UG/NX 虽然都是 CAD 系统，但它们各自的 API 设计哲学差异巨大：

| 维度 | SolidWorks | Creo (ProToolkit) | UG/NX (NXOpen + UF) |
|---|---|---|---|
| API 风格 | COM 接口，面向对象 | 元素树（Element Tree）+ 函数式 | NXOpen 面向对象 + UF 过程式回退 |
| 特征定义 | FeatureData 对象封装全部参数 | 通过 ProElement 树形结构定义 | Builder 模式，Set 方法逐个设参 |
| 终止条件 | 枚举 + 深度值/引用 | 元素路径 + 选择 | EndCondition 对象 |
| 方向表达 | Direction 向量 + ReverseDirection 开关 | 元素树方向字段 + 参考面推断 | Direction 对象 + Sense |
| 坐标系统 | ModelToSketchTransform 矩阵 | ProSection 内建 2D + Placement 矩阵 | Orientation() 矩阵 |

统一层的存在意义就是：**把三套完全不同的 API 表达抽象成一套公共语义，读的时候各自往里填，写的时候各自从里面取。**

### 1.2 阅读方法

每个特征类型按以下结构组织：

1. **统一层字段定义** — `CFeatureXXX` 有哪些字段、类型、语义
2. **读取映射表** — 三系统各自的 API 读取路径和转换计算
3. **创建映射表** — 三系统各自的 API 写入路径和补算推导
4. **异构性分析** — 系统差异如何影响了统一层设计

### 1.3 坐标语义核心约定

这是整个文档最容易混淆的地方，先明确：

- **`sketchCSys`（LocalCSys）**：保存在**世界坐标**中的草图局部框架（Origin/XDir/YDir/ZDir 都是 3D 世界量）
- **草图段坐标**：保存在**草图局部坐标**中（依赖 LocalCSys 解释）
- **旋转轴 `Axis.Direction`**：保存的是**统一后的特征正旋转方向**，不是原始 edge/axis 几何方向
- **拉伸 `Direction`**：保存的是归一化后的拉伸方向向量

### 2.2 共享的 SweepExtent 结构

Extrude 和 Revolve 共用同一套终止条件表达。源码见 `UnifiedFeatures.h:289-312`。

| 字段 | 类型 | 语义 |
|---|---|---|
| `type` | SweepExtent::Type | `VALUE / SYMMETRIC / THROUGH_ALL / THROUGH_ALL_BOTH_SIDES / UP_TO_NEXT / UP_TO_ENTITY / UP_TO_EXTENDED / THRU_POINT / UNKNOWN` |
| `value` | double | 深度值（Extrude 为长度，Revolve 为弧度） |
| `offset` | double | 偏移量 |
| `hasOffset` | bool | 是否启用偏移 |
| `isFlip` | bool | 方向翻转 |
| `isFlipMaterialSide` | bool | 材料侧翻转 |
| `referenceEntity` | shared_ptr\<CRefEntityBase\> | 到面/到点/到实体的引用 |
| `helperPoint` | optional\<CPoint3D\> | 多解消歧辅助点（可选） |

**为什么统一层要定义 `SweepExtent` 而不是保留各 CAD 的终止条件枚举？**

因为三个 CAD 的终止条件概念虽然类似（都可以 Blind、ThroughAll、UpToFace），但枚举值、参数结构、引用方式完全不同。如果不统一，每个特征读写时都要处理三套不同的终止条件表达，逻辑会严重重复和分散。`SweepExtent` 的价值在于：**Extrude 和 Revolve 共用同一个终止条件语义，减少重复映射**。

### 2.3 共享的 ThinWallOption 结构

源码见 `UnifiedFeatures.h:278-282`。ThinWallOption 本身不包含"是否启用"字段——该语义由 `optional<ThinWallOption>` 表达：

| 字段 | 类型 | 语义 |
|---|---|---|
| `startOffset` | double | 起始侧厚度（相对草图轮廓内侧偏置，向内为负） |
| `endOffset` | double | 终止侧厚度（相对草图轮廓外侧偏置，向外为正） |
| `isCovered` | bool | 薄壁端部是否有盖 |

在统一层中，`HasThinWall` 的表达方式是 `optional<ThinWallOption> thinWall` 为空即不启用。

## 3. 草图 (Sketch)

### 3.1 统一层字段定义

```cpp
struct CSketch {
    shared_ptr<CRefEntityBase> referencePlane; // 草图参考平面（引用实体，基类指针可接纳任意引用类型）
    CSketchCSys sketchCSys;         // 草图局部坐标系（Origin + XDir/YDir/ZDir，在世界坐标中表达）
```

`CSketchCSys` 字段：

| 字段 | 类型 | 语义 |
|---|---|---|
| `origin` | CPoint3D | 草图原点在世界坐标中的位置 |
| `xDir` | CVector3D | 草图 X 轴在世界坐标中的方向（归一化） |
| `yDir` | CVector3D | 草图 Y 轴在世界坐标中的方向（归一化） |
| `zDir` | CVector3D | 草图 Z 轴（法向）在世界坐标中的方向（归一化，= XDir × YDir） |

|---|---|---|

**为什么三个系统在段坐标的"原始值"上就不一样？**

根源在于三系统对草图坐标的定义不同：

- **SW**：`GetPoint2()` 返回的是草图坐标系下的 2D 点（Z=0），天然是局部坐标
- **Creo**：ProSection 本质是 2D sketch + placement matrix，section 实体坐标就是 2D 局部坐标
- **UG**：NXOpen 返回的是世界坐标 3D 点，但为了统一层语义一致性，约定按局部语义处理

统一层选择"段坐标 = 草图局部坐标"的约定，是因为重建时所有 CAD 都需要通过 LocalCSys 将局部几何变换到世界坐标来创建，而不是因为三个系统原生都返回局部坐标。

### 3.4 异构性分析

#### 为什么要有 `sketchCSys` 而不是仅靠 `referencePlane`？

三个系统对草图坐标的处理方式不同：

- **SW**：创建草图时会自动关联一个参考面，但实际作图时使用的是 `ModelToSketchTransform` 定义的局部坐标系。不同的草图参考面会导致不同的变换矩阵。
- **Creo**：ProSection 的坐标完全由 `ProSectionLocationGet` 给出的放置矩阵决定，不依赖参考面本身。
- **UG**：`Sketch::Orientation()` 给出方向矩阵，和参考面不完全绑定。

**结论**：仅靠 `referencePlane` 无法唯一确定草图几何在世界坐标中的位置和方向。必须有 `sketchCSys`（LocalCSys）来编码这个"草图平面在世界坐标中的局部框架"，才能在写回时正确还原草图几何。

#### 为什么段坐标不直接保存世界坐标？

如果段坐标直接保存世界坐标，重建时就不需要 LocalCSys 变换了。但问题是：

- **SW 和 Creo 的原生 API** 提取到的段几何本身就是局部坐标（SW 的 `GetPoint2()`，Creo 的 section entity），转成世界坐标需要额外计算
- **多个 CAD 系统共同读写的场景**：如果从 SW 读到的是局部坐标、从 UG 读到的是世界坐标，那统一层就必须选择一种"规范格式"，否则写回时无法知道当前数据是局部还是世界坐标
- **精度问题**：草图局部坐标通常数值较小（接近原点附近），世界坐标可能很大。保持局部坐标可以减少精度损失

**统一层选择"段坐标 = 草图局部坐标"的约定**，使得读侧需要 UG 按约定处理（即使 NXOpen 原生返回世界坐标），写侧全部需要经过 LocalCSys 变换到世界坐标。

#### 三系统在写回时的差异根源

读侧差异主要来自 API 不同但可统一提取，**写侧差异才是真正体现系统异构性的地方**：

1. **SW** 需要在草图编辑模式（`InsertSketch(true)`）下创建段，且 SW 的草图 API 接受世界坐标
2. **Creo** 先建 section 再填充几何，需要对每个段类型做坐标转换和端点吸附（`ApplyEndpointSnap`）
3. **UG** 使用 NXOpen Builder 模式创建，先建 `LineBuilder/ArcBuilder` 设参再提交

这些差异导致每个系统的写回代码路径完全不同，但它们都依赖同一套 LocalCSys → 世界坐标的变换公式。

### 4.4 异构性分析

#### 为什么 direction 不需要单独保存"是否反向"的开关？

三个系统对方向反向的处理方式不同：

- **SW**：有独立的 `ReverseDirection` 属性，默认值为 false，勾选后变为 true
- **Creo**：没有独立的 reverse 开关，方向反向通过将方向向量取反或角度负值来表达
- **UG**：依赖 `Sense`（`SenseForward / SenseReverse`）来控制

统一层的做法是：**direction 向量统一从草图法向推导，反向信息映射到 `extent.isFlip`**。读侧：
- SW 的 `ReverseDirection` → `extent1.isFlip`，**不修改 direction 向量**
- Creo 的 `std_direction_code` → `extent1.isFlip`，direction 统一用草图法向
- UG 可直接读取方向向量

写侧同理，方向向量的正负已经包含了反向信息。

#### 为什么要有 extent1 + extent2 显式表达双向拉伸？

- **SW**：原生支持 `GetEndCondition1/2()`，两个方向独立
- **Creo**：通过 `PRO_E_EXTR_DEPTH_TO` 和 `PRO_E_EXTR_DEPTH_FROM` 分别表达
- **UG**：可能只有一个 EndCondition，也可设置两个方向

统一层始终用 `extent1/extent2` 两个字段来表达双向拉伸，即使只有一个方向有效（另一个的 Type 为 UNKNOWN）。写回时每个系统根据自己的能力决定如何创建。



## 5. 旋转 (Revolve)

### 5.1 统一层字段定义

```cpp
struct CRevolveAxis {
    string referenceLocalID;                    // 轴来自草图段时的 LocalID
    shared_ptr<CRefEntityBase> referenceEntity; // 轴引用（edge/datum axis/sketch seg）
    CPoint3D origin;                            // 轴上一点
    CVector3D direction;                        // 统一语义：特征正旋转方向；ReferenceEntity 保留原始几何方向
};

struct CRevolve : public CProfiledFeatureBase {
    CRevolveAxis axis;             // 旋转轴定义
    SweepExtent extent1;           // 主角度/终止条件（角度值统一使用弧度）
    optional<SweepExtent> extent2; // 第二方向角度/终止条件（可选）
};
```

注意：`CRevolve` 继承自 `CProfiledFeatureBase`，`profileSketchID`、`operation`、`thinWall` 三个字段来自基类。

**`Axis.Direction` 语义特别说明**：这不是原始几何（edge/axis）的方向，而是**统一后的特征正旋转方向**。它满足右手定则：从正方向看过去，旋转角度为正时逆时针旋转。

因为三个系统对"轴方向"和"旋转正方向"的关系定义不同：
- SW 的轴方向 + `ReverseDirection` 开关决定实际旋向
- Creo 的 revolve element tree 有 `FROM/TO` 角度
- UG 的 `Revolve.Builder` 有 `Sense` 控制

统一层必须用自己的规范方向语义，而不是保留任意一个系统的原始方向。

**异构性分析 - 为什么轴方向不能保留原始值？**

三个系统定义"旋转正方向"的方式完全不同：

| 系统 | 旋转轴方向如何定义？ | 反向如何表达？ |
|---|---|---|
| **SW** | 轴对象的几何方向（从草图/边/基准轴来） | `ReverseDirection` 布尔值 |
| **Creo** | 元素树中的轴引用方向 | 角度 FROM/TO 的正负 |
| **UG** | Edge/Datum Axis 的方向 + `Sense` | `Sense.Reverse` |

如果统一层简单地保留原始方向 + 反向开关：
1. 从 SW 读取的 `ReverseDirection=true` 到了 Creo 没有对应的独立开关
2. 从 Creo 读取的角度负值到了 UG 需要转换成 direction + Sense
3. 从 UG 读取的 SenseReverse 到了 SW 没有对应的概念

**统一层的解决方案**：定义一个规范方向（右手定则正旋转方向），把"反向"信息吸收到 `extent.isFlip` 中。这样无论数据来自哪个 CAD，写回其他 CAD 时都遵循：

1. 用 `origin + direction` 定义轴几何
2. 用 `extent1.Value`（正角度）定义旋转范围
3. 用 `extent1.isFlip` 控制是否反向

然后每个系统写回时，再把这个规范语义映射到自己的原生表达。

### 5.4 异构性分析

#### 为什么 `HelperPoint` 不是旋转特征的主字段？

统一层的旋转基本字段是 `origin + direction + extent1/2`，这些已经足以定义一个旋转特征的几何语义。`HelperPoint` 只是在某些终止条件下（如到点终止、多解消歧）才需要的辅助信息。

但三个系统对"消歧"的需求不同：
- **SW** 通常不需要额外的消歧点（通过 selection mark 区分）
- **Creo** 的 `ApplySelectionPoint()` 会从引用中自动推断选择点（face centroid / edge midpoint / axis origin）
- **UG** 最依赖消歧信息：UpToVertex 时需要同时创建 helper datum plane 和 helper point

因此 `HelperPoint` 被设计为可选字段：**不是旋转特征的必要组成部分，而是辅助重建的补充信息**。

#### 为什么角度统一用弧度？

- **SW 原生用弧度**（`GetRevolutionAngle()` 返回弧度）
- **Creo 原生用度**（元素树中是度）
- **UG 原生用度**（NXOpen 接口接受度）

统一层选择弧度，是因为：
1. SW 原生就是弧度，减少 SW 读写的转换
2. 弧度是数学标准单位，在统一层做三角函数运算时不需要额外转换
3. 每个系统在边界层各自做弧度↔度的转换

```
读侧：SW(弧度) → 直接存 ✓
      Creo(度) → 转弧度存
      UG(度) → 转弧度存

写侧：SW(弧度) → 直接传 ✓
      Creo → 弧度转度
      UG → 弧度转度
```

### 6.2 读取映射表 (CAD → UnifiedModel)

#### 6.2.1 method 识别

| SolidWorks | Creo | UG/NX |
|---|---|---|
| **API**: `IRefPlaneFeatureData` 读取原始约束 | **API**: 从元素树推断 | **API**: `nxPlane->Method()` NXOpen 枚举直接读取 |

| 1. `DecodeConstraintType()` 解码原始约束枚举（`feature_datum_plane.cpp:86-113`） | `ToDatumPlaneMethod()` 根据约束类型和数目推断（`read_to_cadexchange.cpp:901-925`） | 枚举值直接映射 |

**SW 为什么需要推理？** SW 的 `IRefPlaneFeatureData` 提供的原始约束是一个整数（`swRefPlaneConstraintType`），其中包含了约束类型和方向信息。需要先解码、再分析约束的组合方式来反推构造方法。

**Creo 为什么需要推导？** Creo 的基准面元素树不直接提供"构造方法"这个值，而是通过一组约束（穿过、偏移、角度等）的组合来实现不同方法。函数名 `ToDatumPlaneMethod()`（注意：`InferPlaneMethod` 是 SW 的函数名，不是 Creo 的）。

#### 6.2.2 referenceEntities / constraints

> **⚠️ 重要说明**：当前三个系统的基准面读取都**不提取 `PlaneConstraint`**，只提取引用实体（`referenceEntities`）。SW 和 Creo 读取路径填充了 `method` 枚举，UG 直接从 NXOpen 读取 `Method()`。`constraints` 字段当前主要用于写回（创建侧指导选择语义），读侧通常为空或只做辅助参考。

| SolidWorks | Creo | UG/NX |
|---|---|---|
| **API**: 从 `IRefPlaneFeatureData` 读取 | **API**: 遍历元素树约束 | **API**: 从特征读取 |
| - 引用面/边/点 → `shared_ptr<CRefEntityBase>` | - 约束类型 → `PlaneConstraintType` | - 引用 → `CRefEntityBase` |
| - 偏移距离 → constraint 参数 | - 引用 → `CRefEntityBase` | - 方法 → `DatumPlaneMethod` 枚举 |
| **转换**: | **转换**: | **转换**: |
| `BuildReferenceFromSwDispatch()` 解析为 `CRefEntityBase` 派生引用 | `DecodeConstraintType()` 映射为 `PlaneConstraintType` 枚举（不产出 `PlaneConstraint` 对象） | `referenceEntities` + `method` 直接映射 |
| 偏移值 → method 推断（`InferPlaneMethod`） | | |

| SolidWorks | Creo | UG/NX |
|---|---|---|

- 同样的"绕轴旋转30度"，在不同的系统中可能旋转方向相反
- 每个系统都有自己的 `ShouldFlipByDefaultDir()` 来判断是否需要取反

### 6.4 异构性分析

#### 为什么基准面要用 method + references + constraints 三段式？

三个系统定义基准面的方式差异巨大：

| 系统 | 如何定义基准面？ |
|---|---|
| **SW** | 使用 `InsertRefPlane` 传入约束类型+值数组（`swRefPlaneOffset` 等），**不是方法枚举**。读侧通过 `InferPlaneMethod()` 从约束反推方法标签 |
| **Creo** | 通过元素树的约束组合来定义（穿过 + 偏移/角度/平行等约束的组合） |
| **UG** | 使用 `DatumPlaneBuilder` 的 `SetMethod()` 枚举 + 引用 + 参数 |

统一层的三段式是为了覆盖三个系统的不同表达方式：
- **`method`** 直接对应 SW 的 `InsertRefPlane` 类型和 UG 的 `DatumPlaneMethod`
- **`references + constraints`** 对应 Creo 的约束组合
- 写回 SW/UG 时以 method 为主，references/constraints 为辅
- 写回 Creo 时以 references/constraints 为主，method 为辅（因为 Creo 需要具体的约束元素）



## 7. 扫掠 (Sweep)

> **当前状态**: 扫掠仅在 SolidWorks 中有完整实现。Creo 和 UG/NX 尚未实现 Sweep 特征的读写。

### 7.1 统一层字段定义

源码见 `UnifiedFeatures.h:362-417`。

`CSweep` 继承自 `CProfiledFeatureBase`，`profileSketchID`、`operation`、`thinWall` 来自基类。

```cpp
enum class SweepProfileKind {
    SketchReference, ///< 引用模型中特征树上的独立草图
    EmbeddedSketch,  ///< 扫掠特征内部截面草绘
    Circular         ///< 参数圆/管截面，不需要独立草图引用
};

struct CSweepCircularProfile {
    double outerRadius = 0.0;
    double innerRadius = 0.0;
};

struct CSweepEmbeddedProfile {
    CSketch sketch;
};

struct CSweepProfile {
    SweepProfileKind kind = SweepProfileKind::SketchReference;
    string sketchID;
    optional<CSweepEmbeddedProfile> embedded;
    optional<CSweepCircularProfile> circular;
};

struct CSweepPath {
    vector<shared_ptr<CRefEntityBase>> references; // 路径引用链（有序）
    optional<CPoint3D> startPoint;                // 路径起点（方向消歧，可选）
    optional<CPoint3D> endPoint;                  // 路径终点（方向消歧，可选）
    bool isClosed = false;                        // 路径是否闭合
};

enum class SweepPathOrientation {
    FollowPath,        ///< 使用各 CAD 默认随路径更新截面方向的实体扫掠行为
    KeepProfileNormal, ///< 尽量保持截面法向
};

enum class SweepSectionPlacement {
    ExistingProfilePlane, ///< 截面来自已有草图平面，可能不垂直路径起点切线
    PathNormalAtStart     ///< 截面位于路径起点法平面
};

struct CSweep : public CProfiledFeatureBase {
    CSweepProfile profile;
    CSweepPath path;
    vector<CSweepPath> guidePaths;
    SweepPathOrientation orientation = SweepPathOrientation::FollowPath;
    SweepSectionPlacement sectionPlacement = SweepSectionPlacement::ExistingProfilePlane;
    optional<double> profilePathAngleCos;
```

### 7.2 读取映射表 (CAD → UnifiedModel)

> **当前三系统状态**:
> - **SolidWorks**: 完整读取和写入（`feature_sweep.h/.cpp`）
> - **Creo**: 未实现
> - **UG/NX**: 已实现 V1 读取和写入（`feature_sweep.cpp`），但有功能限制：仅支持 `SketchReference` 类型 profile，不支持圆管截面（Circular）和嵌入草图（Embedded）；仅读取一条引导线（无多引导线）；orientation 和 sectionPlacement 固定为默认值

#### 7.2.1 SolidWorks 读取映射

| 统一层字段 | SW 读取方式 |
|---|---|
| `profileKind` | `sweepData->get_CircularProfile()` 判断是否圆管；否则 `get_Profile()` 获取 profile -> 区分 Normal/Circular |
| `profileSketchID` | 从 profile feature 提取 sketch ID |
| `path.references` | `sweepData->get_Path(&pathDisp)` 获取整条路径 -> `BuildPathReference()` 解析为引用链 |
| `guidePaths[]` | `sweepData->GetGuideCurvesCount(&count)` + `get_GuideCurves(&guidesVar)` -> 逐个构建 |
| `orientation` | `sweepData->get_PathAlignmentType()` -> `FOLLOW_PATH / PARALLEL_TO_DIRECTION / FIXED` |
| `sectionPlacement` | 从放置语义映射 |
| `thinWall` | 同 Extrude/Revolve 薄壁提取 |

#### 7.2.3 UG/NX 读取映射（V1）

| 统一层字段 | UG 读取方式 |
|---|---|
| `profileKind` | `SweepAlongGuideBuilder->Section()` 读取 section 草图 -> 映射为 `SketchReference`（不支持 Circular/Embedded）（`feature_sweep.cpp:470-476`） |
| `profileSketchID` | 从 Section 解析的 sketch ID |
| `path.references` | `SweepAlongGuideBuilder->Guide()` 读取引导线 -> `TryReadSectionPath()` 解析索引为 `CRefEntityBase` 引用链（`feature_sweep.cpp:478-484`） |

#### 7.2.2 为什么 Sweep 比其他特征更复杂？

Sweep 的复杂性在于**路径和引导线的引用链是"有序的"**，而 Extrude/Revolve 的引用是"独立的"：

| 特征 | 引用特点 |
|---|---|
| Sketch | 一个参考面 |
| Extrude | 一个轮廓 + 可选终止引用 |
| Revolve | 一个轮廓 + 一个轴 + 可选终止引用 |
| **Sweep** | **一个轮廓 + N 个有序路径引用 + M 组有序引导线引用** |

这个"有序"特性使得：
1. 引用链的顺序必须在读取时保持（不能因为 ID 排序而打乱）
2. 写回时要按顺序逐个选中

### 7.4 异构性分析

#### 为什么 Sweep 比其他特征更难跨系统？

| 难度维度 | 说明 |
|---|---|
| **路径引用链** | 三个系统的路径选择语义不同。SW 按段选择，Creo 可能需要 chain/curve，UG 可能需要 curve collector |
| **引导线** | 每个系统对引导线的数量和约束条件不同 |
| **截面方向** | OrientationType 在各系统中的实现方式不同 |
| **圆管参数截面** | 各系统的圆管截面 API 不同 |

这些差异使得 Sweep 的跨系统统一比 Extrude/Revolve 复杂得多，也是目前只有 SW 实现完整 Sweep 的原因。

```
     ↓

     ↓

```

## 9. 总结

### 统一层的核心设计原则

1. **选用最简公分母**：`SweepExtent`, `ThinWallOption`, `ProfileKind` 都是取三个 CAD 系统的交集能力，再补充各自特有能力作为可选字段

2. **规范方向语义**：`Axis.Direction` 不是原始几何方向，而是统一后的规范方向。反向/方向差异通过 `Flip` / `Sense` 等字段吸收，不在方向向量的语义上妥协

3. **坐标分两层**：`sketchCSys`（在世界坐标中的局部框架）+ `segments`（草图局部坐标），写回时通过统一公式还原

4. **辅助信息独立于主字段**：`HelperPoint`、`HelperPlane` 等消歧辅助信息不作为特征主字段，而是写回时的补算层处理

### "统一层"不是什么

- 不是三个 CAD 系统 API 的并集
- 不是三个 CAD 系统 API 的交集
- **而是"写回任何系统都足以重建的公共语义集"**
